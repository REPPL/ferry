# ferry v0.6.0 — adopt the abcd information-architecture standard

**Status:** shipped in v0.6.0. The IA adoption (§§1–9) and fn-6.1 landed; the
fn-4 security work had already shipped in v0.5.3, so it was not re-done, and the
Record folders / `development/` tree were dropped at sign-off. The Phase 5
config-repo sweep is a deferred live-machine follow-up (see the CHANGELOG
migration notes). Sign-off decisions are recorded in `.work/DECISIONS.md`.
**Owner:** REPPL. **Type:** implementation plan (Diátaxis: one plan, no mixing).

This plan supersedes the earlier v0.6.0 draft (public/private `.work/` memory +
date+slug ADRs). It retains that draft's working-memory split, config-repo
agent sweep, and convention-consistency lint verbatim in intent, and **reverses
its single overruled decision**: ADRs adopt sequential MADR `NNNN-title.md`, not
`<date>--<slug>`. The whole abcd information architecture (IA) is folded in as
one coherent release.

## Roadmap (absorbed from the present-state-pass analysis)

A five-agent adversarial analysis of branch `docs/present-state-pass` (a parallel
planning effort expressed as `.flow/specs/fn-1..fn-7` + `.flow/reviews/`)
produced a re-sequenced roadmap for ferry. Its full findings are recorded here so
the `.flow/` specs are no longer needed as a source. The IA adoption described in
§§1–9 of this document is **one item** on that roadmap — re-slotted behind the
security work (see **Sequencing**).

**No double-counting.** fn-3 was split into fn-4 / fn-5 / fn-6 / fn-7, and fn-1 is
superseded by fn-5; the roadmap below counts each concern once under its final
fn-number.

### fn-4 — security patch: credential leak + git RCE (MUST — ships first, standalone)

Real, reproduced credential-leak and RCE paths are open on shipped ferry; the
v0.5.2 secret fix was incomplete. Implement as **extend-not-duplicate** of v0.5.2
(reuse the `gitSync` helper, `splitRef`, the apt `--` guards), and keep **two** git
helpers rather than one — unifying them breaks `gh.GitPush`'s credential-helper flow.

**Eight MUSTs:**

1. **RCE via `[url "ext::sh -c …"]insteadOf`** — executes on sync/fetch against an
   untrusted cloned config repo (CRITICAL, reproduced). Rationale: an attacker who
   controls the config repo gets arbitrary code execution on `apply`.
2. **`git clone <src>` argument injection** — no `--` guard before the source
   argument. Rationale: a crafted source string is parsed as clone flags.
3. **Pre-hardening git runs inside the clone** — `git rev-parse` / `core.fsmonitor`
   run **inside** the freshly cloned repo before it is hardened. Rationale: repo-local
   config executes before defences are in place.
4. **Brewfile content execution** — cask/postflight lines in Brewfile *content*
   execute on `apply --deps`. Rationale: config-repo content becomes code.
5. **Named-prefix token leak** — AWS `AKIA…`, GitHub `ghp_…`, GCP `AIza…`, Slack
   `xox…`, OpenAI `sk-…`, Stripe `sk_live_…` not caught before commit to the shared
   repo. (One of the two v0.5.2 WS2 gaps.)
6. **AWS secret-with-slash misclassified** — a secret containing `/` is treated as a
   filesystem path and skipped. Rationale: real secrets slip the scan.
7. **Indented assignments not scanned** — the secret scan misses `^\s*`-indented
   assignments. (The second v0.5.2 WS2 gap.)
8. **termcfg secrets ungated** — terminal-config secrets are neither rendered nor
   commit-gated the way dotfile secrets are. Rationale: a whole asset class bypasses
   the credential gate.

**fn-4 tier-2 (SHOULD):** block-scalar / heredoc secret detection; structural field
names; `splitRef` traversal hardening; fetch submodule/tag guarding; root apt PATH
seam; `0600` permissions on written secrets; PEM zsh-stop.

**Adversarially REJECTED — do NOT re-raise:** bare `32+`-hex flag; `/`-token split;
bare `sk-` (must anchor `sk-[A-Za-z0-9]{20,}`); a single unified git helper;
`protocol.file.allow=never` (must be `=user`). F19 `dpkg-query --` is defence-parity,
not an open RCE.

Branch spec: `.flow/specs/fn-4-v06x-security-patch-credential-leak-git.md`; findings
register + adversarial notes under `.flow/reviews/`.

### fn-6 — restore hardening (MUST 6.1/6.2 · SHOULD 6.3/6.4a · COULD 6.4b)

- **fn-6.1 (MUST) — leaf-swap TOCTOU close** via `os.Root`, additive to WS1's
  parent-only re-check. Rationale: a symlink swapped in after the parent check still
  redirects a restore write.
- **fn-6.2 (MUST) — restore LOCK.** `restore` takes no lock today and races `apply`.
  Rationale: concurrent restore/apply can interleave and corrupt reconciled state.

The restore EXTRAS (fn-6.3 transactional restore, fn-6.4a last-applied dir-fsync,
fn-6.4b PID-reuse fencing) are deferred — see the pointer below.

Deferred later-band work (fn-5 convergence, fn-2 plugins, fn-7 observability,
restore extras, GNOME/dconf) is planned in
`docs/plans/2026-07-06-v0.7.0.md`.

## 1. Goal and scope

Adopt the abcd IA standard across ferry's two adoption surfaces in a single
minor release:

- **Surface A — ferry's own repository** migrates to the IA: Diátaxis document
  tree, the Record artefact-type layout, MADR sequential ADRs (with the naming
  decision recorded as this repo's ADR 0001), a root `CONTRIBUTING.md`, a
  generated CLI reference under `docs/reference/cli/`, and the committed/private
  working-memory split (`.work/CONTEXT.md`, `.work.local/NEXT.md`).
- **Surface B — what `ferry agents scaffold` emits** for *target* repositories:
  the same working-memory split and any IA layout the scaffold is responsible
  for stamping, driven off the single declarative `scaffoldLayout` table, with
  the golden characterization test updated deliberately.

### Why one coherent release

The working-memory split, the ADR naming, the Diátaxis tree, and the
instruction files that teach all three encode **one convention**. Splitting them
across releases guarantees a window where scaffold output, ferry's own tree, and
the instruction files agents read disagree. Landing them together — gated into
behaviour-preserving and behaviour-changing phases — keeps the convention
consistent and lets the new consistency lint (Phase 6) lock it.

It is a **minor** release: new user-facing scaffold behaviour (the memory split)
plus a new feature (generated CLI reference). Pre-1.0, minor releases may break;
each break is called out in the CHANGELOG **Breaking** section.

### Explicitly out of scope

- **Anything abcd has not finalised.** The Record layout's *placement* and the
  development/work/private *tiering* are under-specified by the IA (see
  Assumptions A1–A2); this plan proposes a concrete baseline and flags the
  choice for sign-off rather than inventing a standard.
- **Relocating dated dev-plans out of a user-visible tree** beyond what the
  Record placement decision (A1) settles. This plan is itself the test case:
  where it lives after A1 is confirmed is noted, not pre-empted.
- **A behavioural eval that grades an agent's instruction compliance** — non-
  deterministic; instruction prose is verified by static consistency + review.
- **ferry's `~/.ssh` and copy-not-symlink invariants** — untouched; no change
  here goes near a trust boundary in the deployed-file path.

## 2. Decisions recorded (with rejected alternatives)

Each decision below is destined for `.work/DECISIONS.md` (one line) and, where it
shapes architecture, an ADR. The ADR-naming decision itself becomes ADR 0001.

- **ADR naming = sequential MADR `NNNN-title.md`.** *Rejected:* `<date>--<slug>`
  (the superseded draft's choice). The IA's sequential MADR wins; date-prefixing
  is reserved for genuinely chronological artefacts (plans, research notes, the
  `DECISIONS.md` log). *Rejected concern that must be mitigated, not ignored:*
  sequential IDs race when two branches each mint "the next" ADR. **Mitigation
  (below) is part of this decision.** Recorded as **ADR 0001**.
- **ADR-race mitigation = optimistic-number + duplicate-`NNNN` lint.** The
  author takes the next free number on their branch; a deterministic lint (added
  to the consistency-lint family, Phase 6) fails CI when two files in a
  `decisions/` directory share an `NNNN` prefix, so a collision *blocks the
  merge* and the second author renumbers. *Rejected:* a central reservation
  registry (ceremony, still races on the registry edit); *rejected:* assign-at-
  merge renaming by a bot (infrastructure ferry does not have). The lint is low-
  ceremony and deterministic; it detects rather than prevents, which is
  sufficient because ADRs merge rarely and the loser's rename is trivial.
- **Working-memory split** (retained from the superseded draft):
  `.work/` (committed) holds `DECISIONS.md` (append-only log) + **`CONTEXT.md`**
  (curated, load-first standing facts). `.work.local/` (git-ignored via
  `info/exclude`) holds **`NEXT.md`** (session handoff, moved here) + `scratch/`
  + `logs/`. *Rejected:* `.work.public/` (redundant suffix); `MEMORY.md` for the
  curated file (collides with Claude Code's machine-local auto-memory).
- **`NEXT.md` in the private layer** — reversible; the shared-handoff pattern
  only pays off at genuine multi-person handoff scale.
- **CLI reference is generated and committed** under `docs/reference/cli/`, with
  a CI currency check (regenerate + `git diff` must be empty). *Rejected:*
  generate-on-demand only (a reader browsing GitHub sees nothing); *rejected:*
  hand-written per-command pages (drift from the Cobra tree). See Assumption A3.
- **Issues graduate into `intents/` + `principles/`; there is no `issues/`
  folder** (LOCKED by the IA). A raised issue that survives triage becomes an
  entry under `intents/` (what we mean to do) and/or `principles/` (a rule we now
  hold), never a standing `issues/` backlog directory.

## 3. Assumptions to confirm at sign-off

These are the points where the IA under-specifies placement. Each has a proposed
baseline; the baseline is what the phases below implement **if confirmed**.

- **A1 — Record placement / tiering.** The IA fixes a flat 7-folder Record
  (`brief/ intents/ principles/ decisions/ roadmap/ plans/ research/` +
  `README.md`) and names the tiering axis `development/ ↔ work/ ↔ .work.local/`,
  but does not say whether the Record lives under `docs/`, under a top-level
  `development/`, or elsewhere.
  - *Proposed baseline (recommended):* realise the Record **under `docs/`**.
    `docs/decisions/`, `docs/plans/`, `docs/research/` already exist there and
    match the current `docs/README.md`; add `docs/brief/`, `docs/intents/`,
    `docs/principles/`, `docs/roadmap/`. `.work/` is the *work* tier (committed
    session memory); `.work.local/` is the private tier.
  - *Cleaner-but-bigger alternative:* a top-level **`development/`** tree holding
    all 7 Record folders, leaving `docs/` **purely Diátaxis** (tutorials/how-to/
    reference/explanation + assets). This realises the tiering literally but is a
    larger migration, moves ADRs/plans/research out of `docs/`, and contradicts
    the current `docs/README.md` and the scaffold's `docs-README.md` template.
  - *Decision needed:* which placement. The baseline keeps the diff small and
    consistent with today's tree; the alternative is a truer reading of the
    tiering and should be a **follow-up release** if chosen, not smuggled into
    this one. ADR 0001's sibling (an ADR recording Record placement) is written
    against whichever is confirmed.
- **A2 — the four new Record folders' content model.** `brief/`, `intents/`,
  `principles/`, `roadmap/` are new to ferry. Proposed: each gets a short
  `README.md` stating its purpose and naming; they start empty otherwise (issue
  graduation, per §2, populates `intents/` + `principles/` over time). Confirm
  whether v0.6.0 seeds any initial content or ships the scaffolding only.
- **A3 — CLI reference committed vs on-demand.** Baseline: committed under
  `docs/reference/cli/` with a currency check. Confirm; if on-demand is
  preferred, Phase 5 drops the committed tree and keeps only `make gen-docs`.
- **A4 — Diátaxis page splits.** Two current pages mix types and must split

  filenames before the `git mv` phase.
- **A5 — `go.mod` impact of the CLI generator.** See §5: importing `cobra/doc`

  module graph, not yet in ferry's `require`). Confirm whether to accept the two

## 5. Cobra → `docs/reference/cli/` generator

**Mechanism.** `github.com/spf13/cobra/doc`'s `GenMarkdownTree(root, dir)` walks
the command tree and writes one markdown page per command. cobra is **already a
direct dependency** (`go.mod`: `github.com/spf13/cobra v1.10.2`).

**`go.mod` nuance — must confirm (A5).** `GenMarkdownTree` lives in the `doc`
package's `md_docs.go`, which imports only stdlib + cobra. But Go compiles a
whole package: the `doc` package also contains `man_docs.go`, which imports
`github.com/cpuguy83/go-md2man/v2` (and transitively `russross/blackfriday`).
Those are in **cobra's** module graph (one `go-md2man` line is already in
`go.sum`) but are **not** in ferry's `require` block today. Importing `cobra/doc`
therefore makes `go mod tidy` add them as **indirect requires** — technically new
`go.mod` entries, so per the repo's ask-first-on-dependencies rule this needs a
maintainer nod. Two clean options:
- **(a) Accept the two indirect requires.** They are graph promotions of modules
  cobra already carries, not genuinely new external code paths pulled from
  nowhere. To keep them out of the *shipped ferry binary*, put the generator in a
  **separate `main` package** (e.g. `tools/gendocs/`, or an `//go:build tools`-
  tagged file) that the ferry binary never imports — only the generator links
  `go-md2man`.
- **(b) Zero new dependency.** Skip `cobra/doc` entirely; hand-roll a ~40-line
  walker over `*cobra.Command` (`Name()`, `Short`, `Long`, `Flags()`,
  `Commands()`) that emits the same markdown. No new module at all; a little code
  to own. Recommended if the maintainer wants strict zero-dependency purity.

**Wiring.**
- Add an exported accessor `cmd.Root() *cobra.Command` (rootCmd is unexported)
  so a generator outside package `cmd` can reach the tree.
- `tools/gendocs/main.go`: call the chosen generator into `docs/reference/cli/`.
- `Makefile`: a `gen-docs` target (`go run ./tools/gendocs`), `.PHONY`.
- **Committed output** (baseline, A3): the generated `.md` tree is committed so
  GitHub browsers see it.
- **Currency check** (analogous to `docs-currency-lint`): a CI step (and
  optionally a pre-commit hook) regenerates into a temp dir and runs
  `git diff --exit-code docs/reference/cli/` — a non-empty diff fails, forcing
  the author to run `make gen-docs` and commit. This closes the drift the
  hand-written `commands.md` currently risks.
- **Overlap with `reference/commands.md`:** the generated tree owns per-command
  detail; `commands.md` becomes a short curated index/overview linking into it.

## 7. Gated phases, ordered by dependency, with STOP conditions

Behaviour-preserving and behaviour-changing work are **separate commits/PRs**
(repo rule). Each phase has its own review gate.

- **STOP conditions (declared up front):**
  - Assumptions A1–A5 are **not** confirmed at sign-off → stop; do not start
    Phase 1. A1 (Record placement) and A5 (`go.mod`) especially gate later
    phases.
  - The golden test changes in a phase labelled *behaviour-preserving* → stop;
    behaviour was not preserved.
  - `make gen-docs` requires a `go.mod` entry the maintainer has not approved →
    stop; fall back to the hand-rolled walker (5b) or defer §5.
  - Any config-repo edit cannot be applied via `ferry apply` (a class isn't
    managed) → stop and report; do not hand-edit deployed copies.

- **Phase 0 — sign-off (this doc).** Resolve A1–A5. No code.

- **Phase 1 — ferry-repo doc migration (behaviour-preserving).** `git mv` the
  Diátaxis pages (§4a-i), split `agents.md`/`ssh.md` (A4), fix all links, rewrite
  `docs/README.md`. Gate: `docs-currency-lint` clean; every internal link
  resolves; no code touched. *No scaffold/emitter change.*

- **Phase 2 — ferry-repo Record + working memory (behaviour-preserving for
  code).** Create `docs/decisions/` + ADR 0001 + ADR 0002; the four new Record

  add `.work/CONTEXT.md`, exclude `.work.local/`, note convention in `AGENTS.md`.

- **Phase 3 — scaffold emitter change (behaviour-changing).** Edit
  `scaffoldLayout`/`scaffoldDocsDirs` + doc comments (§4b); update the golden
  test and `scaffold_test.go` deliberately; add the `CONTEXT.md` template to the
  config repo. Gate: `gofmt -l .` empty, `go vet ./...`, `go test ./...`, and the

- **Phase 4 — CLI-reference generator (new feature).** `cmd.Root()` accessor,
