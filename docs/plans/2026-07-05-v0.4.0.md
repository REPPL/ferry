# Plan: v0.4.0 — release hardening and compatibility contract

Date: 2026-07-05 · Status: approved, awaiting implementation
Executor: a fresh agent session (no prior conversation context is assumed;
everything needed is in this repo, this plan, and the linked docs).
Baseline: `main` after PR #2 (agents domain + asset-mapping registry).
Out of scope: capture-back for agents-domain sources (planned for v0.5.0).

Read first: `docs/agents.md` (the domain this release ships),
`docs/RELEASE.md` (current release flow), `evals/README` (eval contract),
`AGENTS.md` (created in W6 — if present, it is authoritative for commands).

STOP conditions (whole plan):
- STOP if any step would force-push, delete a git tag, or bypass a failing
  check; report instead.
- STOP if the ruleset (W3) and the release flow cannot both hold — do not
  weaken either silently; present the conflict.
- STOP and present options if a state-file migration (W4) cannot be made
  lossless.
Every workstream lands as its own PR off `main`; the full test suite and
eval suite pass at every commit; no AI attribution in commits; docs in
British English, present tense, no private repository names.

## W1 — Release builds the tag, gated by tests

The release workflow checks out the default branch tip; binaries must be
built from the TAG's commit. Restructure into two jobs: `verify` (build,
vet, full `go test ./...`, full eval suite against a built Linux binary —
same steps as ci.yml's `evals-linux`) checked out at `${{ github.ref }}`,
and `release` (`needs: verify`) building from the same tag ref.
- Done when: a tag on an older commit releases binaries built from that
  commit (eval: create a throwaway tag on HEAD~1 in a fork/dry-run and
  assert the stamped version's commit); a red test suite aborts the release
  (eval: intentionally failing test on a scratch tag branch → no release).

## W2 — Provenance: attestations; tags become immutable

Add `actions/attest-build-provenance` (SHA-pinned) over all four binaries
in the release job (`id-token: write`, `attestations: write` job-level).
Document verification in README (`gh attestation verify <binary> -R <repo>`).
Change `scripts/prune-releases.sh`: superseded RELEASES may still be
pruned, but git TAGS are never deleted.
- Done when: `gh attestation verify` succeeds against a released binary
  (eval: run it in CI post-release as a job step); prune dry-run output
  lists releases only, never `git tag -d`/`push --delete` on tags; a
  regression test covers the prune script's tag behaviour.

## W3 — Checksums leave git history; ruleset goes on

Replace the pin-commit flow: `pin-checksums` writes `checksums.txt`
uploaded as a release ASSET (alongside the attestation); `install.sh`
fetches `checksums.txt` from the release it is installing and verifies
binaries against it, still failing closed when absent. The release
workflow then pushes nothing to any branch — remove that step and the
`contents: write`-for-push need it carried.
Then enable the repository ruleset on the default branch: require PR
before merge (0 approvals), required checks `check`, `evals-linux`,
`gitleaks`, `zizmor`, block force pushes, restrict deletions. No bypass
actors.
- Done when: a full release run creates no branch commits (assert in the
  workflow); `install.sh` on a clean container installs and verifies the
  new release end-to-end (extend the existing installer eval); a direct
  push to main and a force push are both refused by the remote (manual
  eval, documented output); CI required checks gate a test PR.

## W4 — Compatibility contract (pre-1.0 housekeeping)

1. Add a `version` field to every persisted state file the domain machinery
   owns (last-applied store, agents target record, backup journal
   manifest). Read path: missing field = version 1, migrate-on-read
   forward-only, pre-migration file backed up via the existing Backuper.
2. Compat fixtures: commit a fixture state directory produced by v0.3.x
   code under `internal/*/testdata/compat/`; tests assert current code
   reads it, migrates it, and `restore` still reverses correctly from it.
3. `docs/reference/compatibility.md`: the three surfaces (CLI, `ferry.toml`
   schema, on-disk state), the pre-1.0 rule (minor releases may break, with
   a CHANGELOG "Breaking" entry), the deprecation contract from v1.0.0
   (old form keeps working for one minor release minimum; warnings name
   the replacement command verbatim so an agent hitting one self-corrects),
   and: defaults never change silently.
- Done when: fixture tests pass; a synthetic future-version state file
  produces a clear refusal, not corruption (test); the doc exists and
  matches behaviour.

## W5 — Version housekeeping

In-source dev version reports the current line (`v0.4.0-dev` once this
plan's work is on main). CHANGELOG.md (Keep-a-Changelog format) covers
v0.4.0: agents domain, asset mappings, scaffold modes, adopt, release/
provenance changes, with a "Breaking" section if W3 changes any installer
behaviour users could observe.
- Done when: `ferry version` on an untagged build reports the current
  line; CHANGELOG renders correctly and mentions every user-visible change.

## W6 — AGENTS.md and the release procedure

Create `AGENTS.md` at the repo root (CLAUDE.md/GEMINI.md as relative
symlinks to it): what ferry is (one paragraph), exact build/test/eval/lint
commands VERIFIED by running them (including single-test and single-eval
invocation), boundaries (copy-not-symlink at $HOME, resolved containment,
`~/.ssh` untouchable, capture never rewrites without accept, repo-
authoritative agents domain), definition of done (build+vet+test+evals,
docs synced, CHANGELOG entry). Keep under ~60 lines; no workspace-level
principles (those load globally).
- Done when: a fresh agent session builds, tests, and runs the eval suite
  using only AGENTS.md; every command in it has been executed successfully.

## W7 — Tag v0.4.0

Preflight: all prior workstreams merged; suite and evals green on main;
CHANGELOG finalised. Tag `v0.4.0`, push the tag, watch the run
(`gh run watch`), then verify: release exists with four binaries +
`checksums.txt` + attestations; `gh attestation verify` passes;
`curl … | bash` install works on a clean environment and `ferry version`
reports `v0.4.0`.
- Done when: every verification above is demonstrated with command output,
  not asserted.

## Sequencing

W1 → W2 → W3 (each depends on the release workflow state left by the
previous), W4/W5/W6 in parallel with any of them, W7 last. If the merged
asset-mapping review (PR #2) surfaces confirmed findings, fixing them is
W0 — it precedes everything.
