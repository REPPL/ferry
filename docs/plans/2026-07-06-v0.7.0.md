# ferry v0.7.0 — config plugins (git, Emacs, iTerm2, Homebrew, npm) on a converged registry

**Status:** signed off — all decisions resolved (§6); implementation-ready and
scheduled to begin with Phase A (fn-5) once credits return. Nothing here is
implemented yet. This revision supersedes the earlier v0.7.0 backlog draft: it
makes the seven config plugins the headline, **reverses that draft's emacs
DROP**, and folds in a five-stream state-of-the-art pass plus a full
code-feasibility map. The deferred later band (restore extras, GNOME/dconf,
machine-readable output) is retained at the end.
**Owner:** REPPL. **Type:** implementation plan (Diátaxis: one plan, no mixing).

## 1. Goal and scope

Carry seven more of a developer's core configurations across machines, each with
the per-machine `.local` overlay and secret handling ferry already gives
dotfiles: **git**, the **Emacs** editor config, **iTerm2**, **Homebrew**,
**npm**, **tmux**, and the **macOS Cocoa key-bindings file**
(`~/Library/KeyBindings/DefaultKeyBinding.dict`). This settles the prior draft's
open decision R2 ("do we want more config plugins?") — the answer is **yes** —
which justifies the **fn-5 domain convergence** the draft gated behind it. Two
observability items (`doctor` invariant checks and a reproducible bundle-SHA) are
also folded in. Per the sign-off, **fn-5 lands first** and every plugin is then
built once on the converged domain interfaces.

Out of scope (deferred, §8): GNOME Terminal/dconf and the restore-hardening
later band. Homebrew `cleanup` is out (install/reconcile-only).

## 2. Findings that reshape the design

A code-feasibility map and a five-stream SOTA pass changed several of the prior
draft's assumptions. These are load-bearing:

- **fn-5 is not implemented — still only planned.** There are no `FileDomain`/
  `ResourceDomain` types; the `isZsh()` oracle (`cmd/apply.go`, `cmd/capture.go`)
  and the hardcoded `{"iterm2","terminal"}` literal lists still drive routing.
  `plugin.Plugin.Deploy()` is dead for deploy routing (only the init wizard reads
  it). So a genuine *config-plugin that must deploy* is blocked on fn-5 — unless
  it reuses an existing non-plugin seam.
- **iTerm2 already exists** as a native ResourceDomain (`internal/terminal/`,
  `defaults export/import`, `NewITerm2`). But the SOTA rejects its
  "load prefs from a custom folder" mechanism for a capture/apply tool (iTerm2,
  not ferry, decides when the file changes; it races hash reconciliation). The
  reviewable representation is **Dynamic Profiles JSON**.
- **Homebrew is already ~90% built** in `internal/deps` (`brew bundle dump` on
  capture, `brew bundle` install, the allow-list gate). A "Homebrew plugin"
  should **extend `internal/deps`**, not be a new `plugin.Plugin`.
- **Emacs fits ferry's existing config-file-*tree* machinery.** `internal/termcfg`
  (and the agents domain) already fan a directory into per-file **nested targets**
  (`dotfile.NestedTarget`) that run the full `~/.ssh`/containment guard. Modelling
  Emacs on that shape needs **no change to the flat `TargetFor` boundary** and
  **no fn-5 dependency** — which removes the exact risk that drove the DROP. The
  DROP assumed the zsh-style parse/split plugin model; the file-tree model is
  available today.
- **`os.Root` (Go 1.24) — the SOTA-recommended safe nested-deploy primitive —
  already landed in ferry v0.6.0** (the fn-6.1 leaf-swap fix). Reuse it for
  writing a config tree into `$HOME` without a swapped intermediate symlink
  escaping it.
- **The maintainer's Emacs config is ideally shaped already.** It is a literate
  config (`init.el` bootstraps `inits/repp.org`, tangled at load) whose
  `.gitignore` already whitelists exactly the carry set, and whose `init.el`
  already loads a machine-generated `inits/custom.el` if present — i.e. the
  per-machine overlay pattern is native. It is not yet deployed to a standard
  Emacs directory, so ferry would establish `~/.emacs.d` as the deploy target.
- **Local reality:** no nvm/fnm and no `~/.npmrc` are present, so npm globals are
  a plain reinstall-from-list and the `.npmrc` token hazard is prospective.

## 3. Per-plugin design

### 3.1 git — `~/.gitconfig` (FileDomain; the fn-5-blocked flagship)

- **Representation:** carry a shared `~/.gitconfig` (aliases, `core.*`,
  `init.defaultBranch`, ergonomics) whose **last** directive is
  `[include] path = ~/.gitconfig.local`. git applies includes inline and
  last-wins, so the machine-local file overrides — a *native* equivalent of
  ferry's overlay (SOTA R7). `[includeIf "gitdir:…/"]` for per-directory identity
  lives in the `.local` layer too (R8).
- **Identity is never shared.** `user.email`, `user.name`, `user.signingkey`,
  `gpg.program`, and `credential.helper` are forced **LOCAL/DROP-only** in
  `Analyze` — a WholeFile deploy that shared them would silently corrupt one
  machine's commit identity with another's. `signingkey` is identity-not-secret
  (a key id / public-key path) and is safe in the local file; `credential.helper`
  must be `osxkeychain`, never `store`, and `~/.git-credentials` is never carried
  (SOTA R9).
- **Secrets:** a **net-new sub-value span extractor** for `url.*.insteadOf` and
  `http.extraHeader` (secrets live inside URL substrings / ordered multivalues,
  unlike zsh's `VAR=value` shapes).
- **Directive trio:** a git-INI `[include]` emit/detect/strip trio parallel to the
  shell-`source` one, placed *after* every existing include/includeIf block.
- **Depends on fn-5:** yes — plugin-owned directive dispatch + registry-driven
  deploy/capture routing (deploy today only knows `isZsh`).

### 3.2 Emacs — `~/.emacs.d/` (config-file-tree FileDomain; reverses the DROP)

- **Representation:** model on `internal/termcfg`, not on the zsh plugin. Map an
  `emacs/` source tree in the config repo to `~/.emacs.d/`, fanned file-by-file
  into `dotfile.NestedTarget` (already containment-guarded) and written through
  `os.Root`. No change to the flat `TargetFor` boundary; no new parser.
- **Deploy model — copy, NOT symlink (D0: adopt, confirmed).** ferry's core
  invariant is that nothing under `$HOME` is symlinked: `~/.emacs.d/` becomes a
  ferry-managed **regular-file copy** reconciled by hash, like every other domain
  — not a symlink to the source tree. The prior `ln -s repo ~/.emacs.d` habit is
  replaced by the apply/capture cycle: edit the config-repo source then
  `ferry apply` (or edit `~/.emacs.d` then `ferry capture`). For this literate
  config that adds an `apply` step between editing `inits/repp.org` and Emacs
  re-tangling it — the accepted trade for cross-machine carry, the `.local`
  overlay, and secret handling. (This account has no Emacs deployed yet, so
  adoption is a clean first `apply`, not a symlink migration.)
- **Carry set** (the maintainer's `.gitignore` already defines it):
  `init.el`, `inits/repp.org`, `docs/`, `README`, `LICENSE`, and `early-init.el`
  if added. **Exclude:** `elpa/ eln-cache/ *.elc` the tangled `inits/repp.el`,
  `auto-save-list/ transient/ url/ network-security.data`, and session state
  (`recentf`, `savehist`, `saveplace`) — SOTA exclude list.
- **Per-machine overlay:** two complementary hooks, both native:
  (a) `inits/custom.el` — the Customize-written machine file — is the `.local`
  overlay (`local/emacs/inits/custom.el`); `init.el` already loads it. (b) an
  optional `init.local.el` loaded last via
  `(load (locate-user-emacs-file "init.local.el") t)` for hand-authored
  machine bits (fonts, `exec-path`, DPI, GUI-vs-tty) — SOTA (a).
- **Deploy home:** `~/.emacs.d` (matches the maintainer's model). Note `~/.emacs.d`
  shadows XDG `~/.config/emacs`; if XDG is ever chosen, apply must assert
  `~/.emacs.d`/`~/.emacs*` are absent or they silently win (SOTA a3).
- **Biggest risk:** none of the DROP's boundary risk (it reuses the nested-safe
  path). The only real choice is the carry/exclude set — already nailed by the
  existing `.gitignore`.
- **Depends on fn-5:** no. Ships on the termcfg-style file-tree today; folds into
  the unified FileDomain when fn-5 lands.

### 3.3 iTerm2 — two artefacts (Dynamic Profiles FileDomain + filtered plist)

Per SOTA, represent iTerm2 as **two** git artefacts, not one plist:

- **Profiles → Dynamic Profiles JSON** in
  `~/Library/Application Support/iTerm2/DynamicProfiles/*.json` — plain,
  live-reloaded, mergeable JSON. `"Rewritable": false` (repo-authoritative:
  edit the repo JSON, not the app — mirrors ferry's agents-domain rule and keeps
  byte-stability); a stable `Guid` per profile is its identity and is **frozen**,
  never regenerated on capture (a changed GUID orphans the deployed profile).
  This is a **termcfg-style FileDomain** (a nested target inside `$HOME`), validated
  with `plutil -lint` before the copy lands (one malformed file disables *all*
  dynamic profiles).
- **Global (non-profile) prefs → a filtered XML plist allowlist.** Capture
  `defaults export com.googlecode.iterm2 - | plutil -convert xml1` then keep an
  allowlisted key set (never the whole plist — `NoSync*`/geometry keys are
  volatile). Apply via `defaults import` **with iTerm2 not running**, then
  `killall cfprefsd`. This extends the existing terminal ResourceDomain but
  **retires its custom-prefs-folder mechanism** (SOTA rejects it).
- **Per-machine divergence:** a `.local` child Dynamic Profile
  (`Dynamic Profile Parent GUID` → the shared parent) overriding only font
  size/GPU/hotkey, plus `TEXT_SIZE_CHANGES_AFFECT_PROFILE = false` so Cmd-+/-
  never dirties the shared profile.
- **Biggest risk:** the global-plist round-trip (iTerm2 rewrites the plist on
  quit; cfprefsd caches it) — **apply must refuse/warn when iTerm2 is running**.
- **Depends on fn-5:** no. Dynamic Profiles is termcfg-shaped; the global plist
  extends the existing ResourceDomain.

### 3.4 Homebrew — extend `internal/deps` (not a new plugin)

- **Already built:** `brew bundle dump` on capture (`ReDumpManifest`),
  `brew bundle` install (`Install`/`installBrew`), and the fail-closed allow-list
  gate (`ValidateBrewfileDirective`). The Brewfile *is* the git-tracked
  representation (SOTA R1).
- **v0.7.0 work:** promote `deps` from behind `--deps` into the default managed
  status surface (a wiring choice in `internal/config/scope.go` `IsManaged` + the
  status/apply call sites) so `ferry status` reports Brewfile drift. Shared
  Brewfile + machine-only casks/`mas` in the `.local` overlay (SOTA R2).
- **Cleanup is OUT (D5, confirmed): install/reconcile-only.**
  `brew bundle cleanup --force` prunes undeclared packages but since 2026
  uninstalls **every** App Store / TestFlight app not in the Brewfile (no
  provenance) — too dangerous for v0.7.0. Not shipped; revisit later behind an
  explicit dry-run-first `--no-mas` flag if ever wanted.
- **Biggest risk:** a *second* Brewfile owner. `internal/deps` must stay the
  single choke point and the allow-list the single gate.
- **Depends on fn-5:** no.

### 3.5 npm — `~/.npmrc` dotfile + a deps-style globals list

- **`~/.npmrc` is a flat single-file dotfile** (`~/.npmrc`), so it fits the
  **existing dotfile domain today** — no fn-5, no nesting. Carry it with the
  registry auth token as a **ferry secret placeholder / `${NPM_TOKEN}` env-ref**
  (npm expands env at read time), rendered on deploy and stripped on capture by
  the mechanism ferry already has (SOTA R6). User-level only; never a repo-level
  `.npmrc`.
- **Global packages → a deps-style list.** Capture `npm ls -g --json --depth=0`
  as **names only** (never versions/tree — churn, SOTA R4), reconcile with
  `npm i -g` from the list. If nvm is ever adopted, carry
  `$NVM_DIR/default-packages` (its native declarative file, R5); fnm has no
  equivalent. This is a new `internal/deps` manager beside brew/apt (note the
  Brewfile gate already *refuses* an `npm` directive, so it must be its own
  manager).
- **Biggest risk:** the `.npmrc` token — routed through the existing
  secret pipeline. (No `~/.npmrc` exists locally today, so this is prospective.)
- **Depends on fn-5:** no (`.npmrc` is a flat dotfile; the globals list is
  deps-shaped).

### 3.6 tmux — `~/.tmux.conf` (FileDomain)

- **Representation:** carry a shared `~/.tmux.conf`; per-machine bits in
  `~/.tmux.conf.local` sourced by a `source-file -q ~/.tmux.conf.local` directive
  (the tmux analogue of zsh's `.local` sidecar). Simplest grammar of the
  plugins — a good early proof of the block model on the converged interface.
- **Secrets:** a net-new sub-value span extractor for quoted `set -g @token '…'`
  values (tmux secrets sit inside a quoted option value, where zsh's whole-line
  fallback would clobber the `set` prefix).
- **Directive trio:** a tmux `source-file` emit/detect/strip trio.
- **Depends on fn-5:** yes (FileDomain plugin with a plugin-owned directive).

### 3.7 macOS key bindings — `~/Library/KeyBindings/DefaultKeyBinding.dict` (FileDomain)

- **Representation:** a **single** old-style (NeXT/ASCII) property-list `dict` at
  a fixed nested path under `~/Library/KeyBindings/`. A single-file FileDomain
  (nested target inside `$HOME`, like a config-file terminal's single file).
- **Round-trip — the cleanest of the seven.** The Cocoa text system only *reads*
  this file at app launch; the OS never rewrites it (unlike a `defaults` app
  plist that cfprefsd constantly reformats). So a hash reconcile shows drift only
  when the user actually edits it — zero churn. No SIP/TCC (plain user-writable
  path). Lowest-risk plugin.
- **No `.local` overlay (D9, resolved).** Content is keyboard *behaviour* —
  machine-agnostic — and the old-style dict has no include/merge mechanism, so an
  overlay could only be a whole-file swap, which earns nothing. Ship it flat; a
  genuinely divergent machine would use a separate top-level file, not a layer.
- **Format hygiene:** validate with `plutil -lint`, but **never `plutil -convert`**
  it — convert emits only `xml1`/`binary1`/`json` and would destroy the readable
  old-style diff. Enforce UTF-8/no-BOM and reject a `bplist00` magic header (an
  editor silently saving binary), e.g. via `.gitattributes text` + a lint check.
- **Apply/reload:** apply is a plain file copy; bindings load at app launch, so
  "reload" = relaunch the affected app (no logout needed). Document this so a
  still-running app showing old bindings isn't filed as an apply failure. Note
  too that Electron/VS Code/Terminal.app honour only a tiny subset — an app
  limitation, not a ferry fault.
- **Do not carry** the SIP-protected system template
  `…/AppKit.framework/Resources/StandardKeyBinding.dict`, and do **not** conflate
  this with per-app `NSUserKeyEquivalents` (a churning `defaults` domain).
- **Secrets:** none.
- **Depends on fn-5:** no inherent coupling (single nested file); built in the
  post-fn-5 phase for a uniform domain surface.
- **Reference for docs:** Apple's archived *Text System Defaults and Key
  Bindings*, plus `github.com/jrus/cocoa-text-system` (the live home of the
  classic write-up — the old `hcs.harvard.edu/~jrus/…` link now 403s) and ss64's
  maintained key-bindings page.

## 4. fn-5 — domain convergence (the enabler)

Converge ferry's extension mechanisms onto **one registry** exposing **two
interfaces** — `FileDomain` (content→target→`ApplyContentDeferred`: dotfiles,
zsh, termcfg, Emacs, git, iTerm2 profiles) and `ResourceDomain`
(`backup.Resource` blob: native terminals, iTerm2 global plist) — and delete the
`isZsh` oracle and the `{"iterm2","terminal"}` literal lists. Freeze the
interface against a **non-zsh FileDomain fixture** (the git `[include]` shape)
before porting, so it fits without a re-cut; land via **byte-stable migration
evals**; and **preserve the two-strip contract** (`stripSourceDirective` vs
`stripFerryOverlayDirective` stay distinct — a guard-bypass eval is required).
XL effort, highest risk on the roadmap; freeze-gated, never big-bang.

**Who needs it:** only **git** is truly fn-5-blocked. Emacs and iTerm2 profiles
are termcfg-shaped today and *absorbed* by fn-5 later; Homebrew and npm-globals
are deps-shaped; npm `.npmrc` is a flat dotfile; the iTerm2 global plist extends
the existing ResourceDomain.

## 5. Sequencing — fn-5 first (D7, confirmed), with STOP conditions

Per sign-off, the domain convergence lands first so every plugin is built once on
the converged interfaces (no build-then-migrate).

- **Phase A — fn-5 domain convergence (the gate).** Freeze `FileDomain` +
  `ResourceDomain` and the resolving registry against a **non-zsh FileDomain
  fixture** (the git `[include]` shape) so the interface fits without a re-cut;
  delete the `isZsh` oracle and the `{"iterm2","terminal"}` literal lists; land
  via **byte-stable migration evals**; preserve the two-strip contract. XL,
  highest risk — reviewed and frozen before any port.
- **Phase B — the seven plugins on the converged interfaces**, easiest→hardest:
  1. **macOS key bindings** (single nested file, no secrets, OS never rewrites it
     — the trivial proof case).
  2. **npm `.npmrc`** (flat dotfile + token secret) and **tmux** (simplest block
     grammar).
  3. **Emacs** (config-file-tree FileDomain; `custom.el`/`init.local.el` overlay).
  4. **Homebrew** and **npm globals** — `internal/deps` extension, independent of
     the FileDomain work; can proceed in parallel with B.1–B.3.
  5. **iTerm2** — Dynamic Profiles FileDomain + filtered global-plist
     ResourceDomain (retire custom-folder).
  6. **git** — the flagship FileDomain, last: hardest secret (URL tokens) and
     identity-never-shared handling.
- **Phase C — observability (independent):** `doctor` invariant checks and the
  reproducible bundle-SHA (both fn-4-only; can land any time).

**STOP conditions:** the fn-5 interface cannot express the git FileDomain without
leaking → re-cut before any port. Any plugin parser fails
`Reassemble(Parse(x)) == x` → stop. Any of git / `.npmrc` / tmux leaks a secret,
or git leaks an identity key, to the shared repo in a capture eval → stop. The
Emacs or key-bindings deploy writes through an unguarded nested path (not
`os.Root`-confined) → stop.

## 6. Decisions to confirm at sign-off

**All decisions resolved.**

- **D0 —** Emacs adopts ferry copy-reconcile (in scope).
- **D1 —** Emacs deploy home is **`~/.emacs.d`** (matches the current config).
- **D2 —** the standalone Emacs config repo's tracked files are **folded under the
  ferry config repo** (`emacs/`); ferry becomes the deploy source of truth.
- **D3 —** Emacs uses **both** overlays: `custom.el` as the `.local` (already
  loaded by `init.el`) for Customize writes, and `init.local.el` (loaded last)
  for hand-authored machine bits.
- **D4 —** iTerm2 pivots to Dynamic Profiles JSON + a filtered global-plist
  allowlist, retiring the custom-prefs-folder path.
- **D5 —** Homebrew is install/reconcile-only; `cleanup` is out.
- **D6 —** npm globals are a plain global list now (no version manager installed);
  an nvm `default-packages` path is added only if nvm is later adopted.
- **D7 —** fn-5 lands first; every plugin is built once on the converged domains.
- **D8 —** tmux and observability (`doctor` + bundle-SHA) are in scope;
  GNOME/dconf and restore-hardening stay deferred.
- **D9 —** the key-bindings file has no `.local` overlay (machine-agnostic; the
  old-style dict format has no merge layer).

## 7. Verification (definition of done, per plugin)

- `make build`, `gofmt -l .` empty, `go vet ./...`, `go test ./...` clean; eval
  suite green with `FERRY_BIN` set.
- Each new parser: a fuzz-proven `Reassemble(Parse(x)) == x` plus positive
  block-boundary fixtures.
- Each secret surface (git URL tokens, `.npmrc` authToken) has a capture eval
  proving the secret never reaches the shared repo and the placeholder never
  deploys literally.
- git: a deploy eval proving identity keys are never Shared across machines.
- fn-5: byte-stable migration evals (zsh deploy/capture/status byte-identical;
  zero spurious de-scope/re-adopt) and a guard-bypass eval for the two-strip
  contract.
- Docs under `docs/` updated; a CHANGELOG entry per user-facing plugin.

## 8. Deferred later band

Scheduled after v0.7.0 (the `doctor`/bundle-SHA half of observability and the
tmux plugin are now in scope, §§1,5):

- **fn-7 `--json` / `list` / `which`:** machine-readable `status`/`diff`/`doctor`
  output pinned by a golden-fixture schema — deferred until fn-5's domain field
  shape has settled in practice, to avoid re-pinning the fixtures.
- **Restore extras:** transactional restore (fn-6.3), last-applied directory
  fsync via an extracted `atomicfile` package (fn-6.4a), PID-reuse fencing
  (fn-6.4b, COULD).
- **GNOME Terminal/dconf:** a dump/load ResourceDomain — the natural first
  non-macOS ResourceDomain once fn-5 lands; open question: profile-UUID strategy.
