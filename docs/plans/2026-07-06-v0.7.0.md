# ferry v0.7.0 — deferred later-band roadmap (convergence, plugins, observability)

**Status:** design backlog, not yet scheduled; version number PROVISIONAL pending
decision R1 (if the security work takes v0.6.0, everything here may shift up a
minor).
**Owner:** REPPL. **Type:** implementation plan (Diátaxis: one plan, no mixing).

This plan collects the later-band roadmap work that the v0.6.0 plan
(`docs/plans/2026-07-06-v0.6.0.md`) defers: domain convergence (fn-5), the
config-plugin series (fn-2), observability (fn-7), the restore-hardening extras
beyond the near-term MUSTs, and the GNOME Terminal / dconf feature. Each item is
recorded here in exactly one place; the v0.6.0 roadmap keeps only the near-term
MUSTs (fn-4 security, fn-6.1 leaf-swap TOCTOU, fn-6.2 restore lock) and points
here for the rest.

The rationale and the adversarially-rejected rules originate from a five-agent
adversarial analysis of a parallel planning effort; the findings are captured
below so no external branch is needed as a source.

## Goal and scope

Land, in dependency order, the deferred work that hardens and extends ferry once
the security release ships:

- **fn-5 — domain convergence:** delete the dead `isZsh` / `DeploySpec`
  deploy-routing oracle and the hardcoded terminal literal lists, replacing them
  with one registry resolving two domain interfaces. This is the enabler for a
  second deployable config plugin.
- **fn-2 — config plugins:** the git config plugin (highest-value second plugin),
  a tmux plugin, and an explicitly-dropped emacs plugin.
- **fn-7 — observability:** doctor invariant checks and a reproducible bundle-SHA
  as the near-term fill-in; `--json` / `list` / `which` deferred as gold-plating
  that itself needs fn-5.
- **restore extras:** transactional restore (fn-6.3), last-applied directory
  fsync (fn-6.4a), and PID-reuse fencing on the lock (fn-6.4b) — the restore
  work *beyond* the near-term fn-6.1/fn-6.2 MUSTs, which stay in v0.6.0.
- **GNOME Terminal / dconf:** a dump/load bridge for the one terminal family
  ferry cannot treat as a config-file copy.

**Out of scope here:** fn-4 (security) and the fn-6.1/fn-6.2 restore MUSTs — those
are near-term work and live in the v0.6.0 plan. This document is the later band
only.

## fn-5 — domain convergence (SHOULD if we want more plugins · else COULD)

**What.** Converge ferry's extension mechanisms onto ONE registry exposing TWO
interfaces, delete the `isZsh()` deploy-routing oracle and the hardcoded
`[]string{"iterm2", "terminal"}` literal lists, and give fn-2 the base it needs.
Reading the code, dotfiles, agents, and termcfg already share the deploy core
(`dotfile.ApplyContentDeferred`); native terminal-preference domains are
genuinely different machinery (`backup.Resource`, shelling out to `defaults`
against an opaque domain id, no `$HOME` path, blob-restore). The two interfaces
reflect that split:

- **FileDomain** — `content -> Target -> ApplyContentDeferred` (dotfile, agents,
  termcfg): overlay kind, domain-owned directive emit/detect/strip, claimed-name
  set, per-name sidecar derivation, secret-rendering flag.
- **ResourceDomain** — `backup.Resource` plus apply-closure and capture-blob
  (native terminals): Backup/Restore/Apply, prefs custom-folder staging,
  control-key strip.
- **One registry** resolves BOTH for routing, scope, `list`, `which`, and the
  `isZsh` deletion. It unifies *resolution*, not deploy mechanics — no
  god-abstraction, no plist domain leaking file-domain no-ops.

**Why.** Exactly one config plugin (zsh) exists today. The convergence only earns
its risk once a second plugin is actually wanted — which is why it is a SHOULD if
decision R2 says yes to more plugins and a COULD otherwise.

**Size / risk.** XL effort, highest risk on the roadmap, and the biggest collision
surface with any in-flight migration (it touches the shared `apply`, `capture`,
and `status` files). It is gated behind a freeze and lands via byte-stable
migration evals — **never big-bang**. The staged shape:

- **Freeze the interface(s) before porting.** Define FileDomain + ResourceDomain
  and the resolving registry, proven against a NON-zsh fixture FileDomain (an
  `init.local.el`-shape sidecar with a different directive) so the interface fits
  fn-2 without a re-cut. Frozen and reviewed before any port begins — if one
  FileDomain interface cannot express both zsh's `<bare>.local` sidecar AND the
  fixture's shape without leaking, the split is re-cut before the ports start.
- **Delete `isZsh` + port routing.** Full seam inventory (the `status.go` literal
  switch, the `restore.go` terminal-domain-id / register pair, the second
  `apply.go` literal); resolve every deploy/capture/status/de-scope site from the
  registry. Behaviour-lock: zsh deploy/capture/status byte-identical.
- **Port the four domains** (zsh as a full-pipeline FileDomain; termcfg and agents
  as FileDomains; native terminals as a ResourceDomain iterating the registry). A
  migration eval applies a prior-release repo, upgrades, and asserts zero spurious
  de-scope / re-adopt — last-applied keys and `local/<domain>/` layout byte-stable
  or a one-time migration provided.
- **Deploy-path fixes on the unified path** (F15 load-preserve `managed=true` on
  re-init; guided risk parity in diff/dry-run and `--force` overwrite reporting;
  additive local dotfiles list).

**Preserve the TWO-strip contract.** `stripSourceDirective` (capture-side) and
`stripFerryOverlayDirective` (guard-side, the empty-over-substantial data-loss
guard) are distinct — do NOT unify them: a domain failing to strip its own
directive would smuggle an empty shared source past the guard, a data-loss
regression. A guard-bypass eval is required.

**Dependencies.** Depends on the patched tree (fn-4) and the hardened write
primitives (fn-6.1/fn-6.2). Gated on decision R2 (do we want more plugins?).

## fn-2 — config plugins (git SHOULD, gated on fn-5 · tmux COULD · emacs DROP)

Three candidate compiled-in config plugins, each mirroring the zsh layout
(`internal/plugin/zsh/{zsh,parse,analyze,starter}.go` plus tests) and consuming
the interface fn-5 freezes. All deploy **Sidecar** (the per-machine local layer is
the point of the product). Parsers are byte-level: `Reassemble(Parse(x)) == x` is
the fuzz-tested invariant, and each ships positive block-boundary fixtures because
fuzz proves round-trip but cannot prove classification.

- **git plugin (SHOULD, gated on fn-5).** Manages `~/.gitconfig` via an
  `[include] path = ~/.gitconfig.local` directive placed *after* every existing
  include / includeIf block (git config is last-wins). This is the highest-value
  second plugin: a WholeFile deploy would let one machine's `user.email` /
  `user.signingkey` overwrite another's — silent identity corruption — so the
  identity keys (`user.email`, `user.name`, `user.signingkey`, `gpg.program`) and
  `credential.helper` offer LOCAL/DROP routes only, never shared. It also detects
  credential leaks in `url.*.insteadOf` and `http.extraHeader` via a net-new
  sub-value span extractor (git secrets live inside URL substrings and across
  ordered multivalued keys, unlike zsh's `VAR=value` shapes). *Why gated:* it
  needs fn-5's convergence first, and the guard must dispatch strip-recognition by
  each target's owning plugin.

- **tmux plugin (COULD).** Manages `~/.tmux.conf` via `source-file -q
  ~/.tmux.conf.local`. Nice-to-have; the simplest grammar of the three, so it
  doubles as the early proof point for the block model. tmux secret spans are
  net-new too (quoted values of `set -g @token '...'` lines, where zsh's
  whole-line fallback would clobber the `set` prefix).

- **emacs plugin (DROP).** Managing `~/.emacs.d/init.el` forces nested-path
  support that changes `dotfile.TargetFor` and `validateHomeTarget` — the
  traversal and `~/.ssh` security boundary shared by *every* domain. Alone that is
  a risky boundary change for scope-creep value, so drop it unless explicitly
  wanted.

**Size / risk.** Medium per plugin; the risk concentration is the emacs
boundary change, which is why emacs is dropped rather than merely deferred.

**Dependencies.** The whole series depends on fn-5's frozen, delivered interface
(plugin-owned directives, claimed-name set, per-name sidecar derivation, and
registry-driven apply *and* capture routing). If any of that is missing, fn-2 is
blocked on fn-5.

## fn-7 — observability (doctor + bundle-SHA SHOULD · `--json` / list / which defer)

Most of this is NOT actually blocked by the convergence — only the JSON-schema
piece needs the converged shape to avoid a schema-churn redo.

- **doctor invariant checks (SHOULD, fn-4-only).** Extend `ferry doctor` to flag
  a missing `config.toml`, a repo path that does not exist or whose `LoadScope`
  fails to parse, a stale apply lock, unknown `[manage]` domains (a typo warning),
  and per-domain (agents/termcfg/terminal) config errors — each with the
  corrective command named. Cheap operational assurance with no new consumer.
- **reproducible bundle-SHA (SHOULD, fn-4-only).** Compute the bundle member mode
  from the git index (`ls-files --stage`) rather than the live filesystem, so two
  exports of identical git-tracked content produce an identical SHA across a
  `chmod`. Self-contained, high-signal: it proves the bundle trust-anchor is a
  pure function of tracked content.
- **F15 `managed=true` fill-in (SHOULD).** `init --github` currently drops
  `managed=true`; load-preserve it on re-init. (This overlaps the fn-5 deploy-path
  fixes; whichever lands first carries it.)
- **guided-apply post-upgrade friction (E10, SHOULD, fn-4-only).** A file whose
  hash matches but has no snapshot is classified `risky`, so a clean file triggers a
  spurious guided prompt after a store-format upgrade. Distinct from fn-5's
  byte-stable migration (that concerns de-scope/re-adopt; this is guided-apply
  classification). Independent.
- **agents capture-back + secret-rendering (E7, SHOULD, fn-4-only).** A
  `SourceCombined` agent harness cannot be captured back to a per-harness local, and
  agents content is never secret-rendered — agents is the one deploy domain still
  bypassing the secret pipeline (the v0.5.3 fn-4 work gated termcfg, not agents), so
  the secret-rendering half is security-adjacent. Independent.
- **shell completions + verbosity flag (E9, SHOULD/COULD).** Cobra completions plus
  a global `-v/--verbose` / `--quiet` levelled logger for refusal/skip messages (the
  acceptance test pins a named refusal and skip message at asserted levels).
  Near-free with cobra; independent.

- **`--json` / `list` / `which` (COULD/defer — needs fn-5).** Machine-readable
  `status` / `diff` / `doctor` output pinned by a golden-fixture schema contract,
  plus `ferry list` (in-scope domains/targets) and `ferry which <name>` (source
  resolution). This is the only piece that hard-depends fn-5: serialising the
  status/diff JSON schema *after* the domain field shape settles avoids re-pinning
  the golden fixtures. Gold-plating with no consumer today — deferred.

**Size / risk.** Small–medium; low risk. The doctor and bundle-SHA pieces can land
independently of everything else.

**Dependencies.** doctor and bundle-SHA depend only on fn-4. `--json` / `list` /
`which` depend on fn-5.

## Restore extras (SHOULD 6.3 · SHOULD 6.4a · COULD 6.4b)

These are the restore-hardening items *beyond* the near-term fn-6.1 (leaf-swap
TOCTOU close via `os.Root`) and fn-6.2 (restore lock) MUSTs, which stay in the
v0.6.0 plan.

- **fn-6.3 — transactional restore (SHOULD).** Reuse the existing journal
  (`journal.go` already has Begin/Commit, an fsynced COMPLETE marker, and
  `RollbackIncomplete`) — do NOT invent a second marker. The interaction to settle:
  a restore-as-journal-run would make the NEXT apply's `RollbackIncomplete` UNDO
  the user's restore, so restore runs are TAGGED and apply's rollback skips them;
  restore relies on its existing snapshot plus a restore-specific completeness
  flag that apply ignores. Acceptance: a crash mid-restore is detected and is NOT
  undone by the next apply.
- **fn-6.4a — last-applied directory fsync (SHOULD).** Route the dotfile
  last-applied `save()` through an extracted `AtomicWrite`/`syncDir` primitive (a
  leaf package such as `internal/atomicfile`, imported by both `backup` and
  `dotfile`, to avoid a `dotfile -> backup` edge that fn-5's convergence could turn
  into a cycle) so the directory is fsynced. Acceptance: the last-applied record
  never lags the deployed file across a crash.
- **fn-6.4b — PID-reuse fencing (COULD/defer).** Record `(pid,
  startTimeUnixNano)` on the lock and read `processStartTime` per-platform (linux
  `/proc`, darwin sysctl, an `_other` stub falling back to `kill(0)`), so a lock
  survives PID reuse (a start-time mismatch marks it stale).

**Size / risk.** fn-6.3 is medium (journal-interaction design is the tricky part);
fn-6.4a is small; fn-6.4b is small but low-value, hence COULD. Tests must be
deterministic (static pre-swap or a package-var seam) — goroutine-timing race
tests are forbidden in acceptance.

**Dependencies.** Depend on the fn-6.1 write primitive being frozen first (fn-6.4a
consumes the extracted atomicfile package). All ride on the patched tree (fn-4).

## GNOME Terminal / dconf (feature)

**What.** Add GNOME Terminal support. Unlike the config-file terminals shipped in
v0.5.0 (a file copy reconciled by hash), GNOME Terminal stores its profiles in
dconf, so this is a **dump/load bridge**, not a file copy: capture dumps the
relevant dconf subtree, apply loads it back. It fits the ResourceDomain shape
fn-5 introduces (backup/restore of an opaque blob against a domain key with no
`$HOME` path), so it is a natural first non-macOS ResourceDomain once fn-5 lands.

**Why.** GNOME Terminal is the remaining common Linux terminal family ferry cannot
carry, because it is the one that is not a plain config file. It rounds out the
config-file terminal support from v0.5.0.

**Size / risk.** Medium. The risk is the open design decision below, not the
plumbing.

**Dependencies.** Best built on the ResourceDomain interface from fn-5; can be
prototyped independently but should not ship its own bespoke routing.

## Sequencing and dependencies

The recommended order, respecting the dependency chain:

1. **fn-6.1 / fn-6.2 restore hardening (near-term, v0.6.0).** Leaf-swap TOCTOU
   close and the restore lock land first and freeze the hardened write primitives.
   These are v0.6.0 MUSTs, listed here only to show the chain root.
2. **fn-5 — domain convergence.** Builds on the hardened primitives; gated on the
   more-plugins decision (R2). Highest risk — freeze-gated, byte-stable migration
   evals, never big-bang.
3. **fn-2 git plugin.** Lands on fn-5's frozen interface; tmux is a COULD
   follow-on; emacs is dropped.

Work that can land any time (fn-4-only, not blocked on fn-5):

- **fn-7 doctor invariants and reproducible bundle-SHA** (and the F15 fill-in).
- **restore extras fn-6.3 / fn-6.4a / fn-6.4b**, once fn-6.1's primitive is frozen.

Work that hard-depends fn-5: fn-7's `--json` / `list` / `which`, and (cleanly) the
GNOME/dconf ResourceDomain.

## Open decisions

- **R2 — Do we want more config plugins (git / tmux)?** This single yes/no decides
  whether fn-5 domain convergence is a SHOULD (needed to unblock the second
  plugin) or a COULD (deferred). It is the gate for the whole fn-5 → fn-2 chain.
  (R1, the versioning decision that governs this document's provisional version
  number, lives in the v0.6.0 plan's Open questions.)
- **dconf profile-UUID strategy.** GNOME Terminal keys profiles by UUID under the
  dconf path. The open question is how ferry carries them across machines: preserve
  the source UUIDs verbatim (simplest, but collides if the target already has a
  profile at that UUID), remap to fresh UUIDs on load (avoids collision but must
  rewrite the profile list and default-profile pointer), or key profiles by a
  stable ferry-assigned name and reconcile UUIDs at apply time. This must be
  settled before the dump/load bridge is built.
