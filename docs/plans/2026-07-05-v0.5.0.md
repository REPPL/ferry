## Theme

v0.5.0 turns `apply` from a terse reconciler into a guided, non-expert-first
walkthrough, and closes the agents-domain loop so live edits flow back to the
repo. It also begins Linux terminal coverage. The three workstreams share one
new piece of state, so that lands first.

## Foundation — the per-target deployment baseline

`apply` records a content snapshot (or hash) for every target it deploys: the
"last-deployed baseline". It is the load-bearing primitive for two features:

- **Capture-back** reads it as the common ancestor to classify a target as
  repo-changed, locally-changed, or both (true divergence).
- **The risk gate** reads it to know a file is "locally modified" — on-disk
  content differs from what ferry last wrote.

Decisions:
- **First apply after upgrade (no baseline yet): confirm every target once.**
  The first post-upgrade `apply` walks every target through the risk gate to
  establish baselines by explicit confirmation. Chosen over snapshotting the
  current on-disk state (which would silently bless a pre-upgrade local edit)
  and over assuming in-sync-with-repo (noisy on drifted machines).
- This adds on-disk state → a state-file version bump and a lossless
  migration, handled the way v0.4.0's W4 handled its migration (write-once
  sibling backup; never routed through the Backuper). Where the baseline
  physically lives (extend the state file vs a sidecar) is an implementation
  choice for the executor; it must migrate losslessly or STOP.

## W1 — Guided `apply` (the first-run/walkthrough headline)

`init` shrinks to setup — clone/create the repo, write config — then hands off
to `apply`, which owns the single walkthrough. There is one guided flow, in
`apply`; `init` keeps only its existing zshrc-adoption step for setup.

Decisions:
- **UI: reuse the existing TUI** (the same component `init`'s adopt wizard
  uses) for visual and behavioural consistency.
- **Trigger: on every apply that has changes.** A clean, in-sync apply prints
  one line (e.g. "in sync, N targets"); it does not walk.
- **Depth: grouped by domain** (dotfiles / agents / dependencies). The user
  confirms a domain wholesale or drills into it.
- **Detail view actions per item:** show full diff · apply or skip this run ·
  route shared vs `.local` · skip-always (remembered).
- **Quiet when safe, stop when risky.** Safe changes auto-apply with a
  one-line summary. A change is *risky* — and halts for confirmation — when it
  would: overwrite a locally-modified file · introduce a brand-new target ·
  remove anything · touch secrets or `~/.ssh`-adjacent paths.
- **Audience: non-expert first**; experts opt out with `--skip-wizard`.
- **Non-interactive + risky pending: fail closed** — exit non-zero and list
  what needs a human. `--skip-wizard` suppresses the UI but still refuses
  risky changes; nothing risky happens unattended.
- **Exclusions** ("skip always" and per-machine opt-outs) live in the existing
  gitignored `.local` layer — one mechanism, consistent with how `.local`
  already wins.

## W2 — Capture-back for agents-domain sources

Live edits to deployed agent files (a project's `CLAUDE.md`, `AGENTS.md`,
skills, hooks) flow back into the config repo instead of being reported as
drift forever. The agents domain stops being strictly repo-authoritative for
its deployed copies.

Decisions:
- **Mechanism: through `ferry capture`** — the same approve/route (shared vs
  `.local`) flow as dotfiles. Explicit and user-initiated; no daemon, no
  file-watcher.
- **Scope: any deployed copy ferry tracks** (per its own state/journal),
  wherever it landed — not limited to the devtree.
- **Divergence (deployed file and repo source both changed since baseline):
  refuse and show a diff.** Never auto-merge, never guess a winner.
- **New files: offer to adopt.** `capture` notices new agent-shaped files in
  tracked projects that ferry never deployed and offers to bring them under
  management. "Agent-shaped" is defined by the **agents-registry patterns**
  (the existing harness/asset registry is the single source of truth; it grows
  as the registry grows) — not a hand-maintained filename list.

Invariant preserved: capture never rewrites a managed file without an explicit
accept, and never auto-commits or pushes.

## W3 — Linux terminal-emulator support

The core is already CI-tested on Linux; this begins terminal coverage there.

Decisions:
- **Config-file terminals first** (Alacritty, kitty, WezTerm) — plain-text
  config files carried like dotfiles. GNOME Terminal (dconf/gsettings) is
  deferred to a later release; it needs a dump/load bridge, not a file copy.
- **Discovery: a built-in registry of known terminals** (terminal → config
  path), enabled per the terminals you use — mirroring how the agents registry
  works today. Chosen over manual manifest entries (no coverage out of the box)
  and over auto-detecting installed terminals (probes the system).

## Sequencing

1. **Foundation** (deployment baseline + state migration) — everything else
   depends on it.
2. **W1** (guided apply) — consumes the baseline via the risk gate; highest
   user-visible value.
3. **W2** (capture-back) — consumes the baseline for divergence detection.
4. **W3** (Linux terminals) — independent of the baseline; can proceed in
   parallel once the terminal registry shape is agreed.

## Open questions for the executor

- Physical home and format of the deployment baseline, and the exact migration
  step (must be lossless or STOP).
- Whether the risk gate's "locally modified" check compares full content or a
  stored hash, and how it treats whitespace/line-ending-only differences.
- Terminal-registry entry shape and how scope and the `.local` layer apply to
  terminal config (per-machine colour schemes are a likely `.local` case).
- Migration UX for the "confirm every target once" first run so it reads as a
  one-time bootstrap, not a scary full-machine prompt.
